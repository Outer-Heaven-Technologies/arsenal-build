# Ingest-from-docs Subagent Prompt Template

Used by `arsenal-build:setup` Step 3B — Path B (ingest user-pointed source docs). Dispatch one subagent per target artifact (FEATURES.md / UX.md / DESIGN.md). The subagent reads user-pointed source documents, extracts relevant content, and writes one lightweight planning artifact at the canonical path with a `Generated by setup` banner.

**One dispatch = one artifact.** Don't combine multiple targets in a single dispatch.

Spawn with this prompt:

```
## Your Role
You are generating a LIGHTWEIGHT planning artifact for the arsenal-build pipeline by extracting relevant content from user-provided source documents. The output replaces what the corresponding `arsenal-planning:<skill>` would have produced — but lighter (no deep interviews, no structured research). The downstream build pipeline reads your output as-is.

## Target Artifact
[FEATURES.md | UX.md | DESIGN.md]

## Canonical Output Path
[.arsenal/FEATURES.md | .arsenal/design/UX.md | .arsenal/design/DESIGN.md]

If the project uses split-mode features (`.arsenal/features/` directory), produce `.arsenal/features/README.md` plus one file per major feature instead of a single FEATURES.md.

## Project Type
[UI (web-marketing) | UI (web-app) | non-UI]

## Source Documents
[List of paths the user pointed at — PRD, Notion export, hand-written spec, brand reference, etc. May also include pasted content. Read every source.]

## Existing Context (read-only)
[Paths to any planning artifacts that already exist at canonical locations (e.g., MVP_SPEC.md if present, or another generated artifact if a previous dispatch already ran). Use as context; do NOT overwrite anything other than your target.]

## Your Job
1. Read every source document in full.
2. Extract the content relevant to the target artifact (see below for what each target needs).
3. Write the target artifact at its canonical path with the banner.
4. Report DONE with: target path, source paths cited, gaps you couldn't fill from the sources.

## Required banner (verbatim, top of file)

```markdown
> **Generated by `arsenal-build:setup` — lightweight bootstrap.**
>
> Source: <comma-separated list of source paths>
>
> This is a lightweight version produced to unblock the build pipeline. For deeper analysis, run `arsenal-planning:<features|ux-web|ux-app|design>` and overwrite this file. The downstream build pipeline reads this file as-is regardless of how it was generated.
```

## What each target needs

### FEATURES.md
A flat list of 3–8 features the product needs at launch. Each feature is one section:

- **Feature name** (short, kebab-case as slug; readable name as heading)
- **One-line description** — what it is from the user's perspective
- **User flow** — 3–5 step happy path (no edge cases yet)
- **Acceptance criteria** — 2–4 testable bullets
- **Data** — 1–2 sentences on what the feature reads / writes / mutates
- **States** — list of visual/behavioral states (default, loading, empty, error, success)
- **Anti-patterns** — what NOT to build into this feature (pulled from source docs if mentioned)

Extract these from product spec docs, PRDs, feature lists, user stories. If the source is sparse, write what's defensible from the source and mark gaps with `<!-- gap: <what's missing> — run arsenal-planning:features for depth -->`.

### UX.md (UI projects only)
Page/screen inventory with section-level structure.

- **Sitemap** — flat list of pages (`/` `/about` `/dashboard` etc.) with one-line purpose each
- **Per-page structure** — for each page, list its major sections (hero / feature grid / FAQ / footer; or app shell / main panel / inspector etc.)
- **Primary CTA** per page (for marketing) / **Primary action** (for app screens)
- **Navigation pattern** — top nav / sidebar / tabs / etc.
- **Components needed** — flat list of components named across pages (Button, Card, Modal, Form, Hero, Footer, etc.)
- **Empty states / error states** — list per page if covered in source

Extract from sitemap files, wireframes, design briefs, page outlines. If the source has visuals but no text inventory, name what's visible in the visuals.

### DESIGN.md (UI projects only)
Lightweight brand spec. Body of 6–8 sections at most.

- **Identity** (1–2 sentences) — what the brand stands for, in plain language
- **Voice** (3–5 bullets) — tone words; example phrasings; what we don't say
- **Color palette** — 3–6 named colors with hex values + role (canvas / ink / brand-primary / brand-accent / surface-elevated / etc.)
- **Typography** — 1–2 typefaces with their roles (display / body / mono); type scale (3–5 steps)
- **Spacing rhythm** — base unit + 4–6 named steps
- **Motion** (1 paragraph) — how things move (snappy / restrained / cinematic / etc.); reduced-motion behavior
- **Inspiration / references** — list of URLs or named brand references from the source if any
- **Optional**: components catalog if the source mentions specific component patterns

Extract from brand briefs, mood boards, design refs, inspiration lists. If the source is "make it feel like Linear and Vercel" with no concrete tokens, infer conservative tokens from named refs and mark them inferred.

## Boundaries
- **Do not modify code.** Your only writes are to the canonical artifact path.
- **Do not write multiple artifact files** in this dispatch. One target only.
- **Do not invent content the sources don't support.** Where the source is silent, leave a gap marker. The user resolves gaps either by adding source detail and re-running setup, or by running the corresponding arsenal-planning skill.
- **Do not deepen beyond the source docs.** No new research, no web fetches to fill in missing brand context. This is a lightweight transformation, not a full spec.
- **Do not skip the banner.** The banner is the contract that signals "lightweight" to future readers.
- **Preserve user voice.** If the source has specific phrasings, anti-startup-speak hard rules, or named guardrails, carry them forward verbatim. Don't paraphrase.

## Output format

After writing the file, return:

```
Status: DONE
Target: <canonical path written>
Sources: <comma-separated paths cited in the banner>
Gaps: <list of `<!-- gap: ... -->` markers added, or "none">
Recommended next step: <"run `arsenal-planning:<skill>` for depth" — name the specific skill>
```

If the source docs are insufficient to produce even a minimum-viable artifact, return:

```
Status: INSUFFICIENT_SOURCE
Target: <canonical path>
Sources read: <paths>
What's missing: <what couldn't be covered from the sources>
Recommended path: <"Path C: intake interview" or "supply additional source docs">
```

The controller will route accordingly.
```

## Controller-side rules

- **One dispatch per artifact.** If the user assigns Path B to both FEATURES and UX, that's two dispatches.
- **Do not re-read the subagent's output.** Confirm the file exists via `ls`; trust the subagent's DONE report. Re-reading defeats the fresh-context benefit and inflates the controller's context.
- **Surface gaps in the final report.** Any `<!-- gap: ... -->` markers should appear in the Step 7 review so the user knows what's incomplete.
- **If a subagent returns INSUFFICIENT_SOURCE**, ask the user: supply more docs (re-dispatch B) or switch to Path C for this artifact (intake interview, then dispatch the bootstrap subagent).
